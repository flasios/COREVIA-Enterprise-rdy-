Section 01

Architecture Principles

Non-negotiable invariants governing every decision across the platform.

Governance-First

No AI reasoning executes before policy clearance at Layer 3. Every request passes through policy enforcement before any engine is invoked.

Append-Only Decision Object

The DecisionObject is immutable. Each layer appends its section. Full audit trail from intake to memory — no mutation, no deletion.

No Side-Effects Before L7

Only Layer 7 and above may persist to external systems. Layers 1–6 are pure read-analyze operations.

Classification Drives Everything

Data classification at L2 determines engine routing, redaction mode, HITL requirements, allowed regions, and retention period.

Sovereign Data Never Leaves

SOVEREIGN-classified data can never route to external engines. Enforced at L2, Engine Router, and IPLAN validation.

No Learning From Rejected

Engine C only ingests APPROVED decisions. Rejected or draft outputs never enter the distillation pipeline.

Domain-Driven Bounded Contexts

13 bounded contexts with clean APIs. Domains own their data, routes, application logic, and infrastructure adapters.

Tenant Isolation by Default

Every data operation is scoped by organizationId. Row-level security policies in PostgreSQL enforce tenant boundaries.

L1

Intake

Normalize raw input into canonical IntakeData
Deep-trim strings, strip nulls, capture serviceId, routeKey, userId, timestamp. Source: brain/layers/layer1-intake/index.ts
L2

Classification

Detect data sensitivity and set security constraints
Keyword scan → Sovereign / Confidential / Internal / Public. Sets: allowCloudProcessing, allowExternalModels, requireHitl, allowedRegions, maxRetentionDays. Detects sector (government, healthcare, finance, defense, education, infrastructure).
L3

Policy Ops

Policy-as-code enforcement gate
4 built-in policies (classification access, budget threshold, sovereign protection, service-specific). Registry packs from DB. Authority matrix: CISO+CTO+Minister → auto-approved based on risk × budget.
Can BLOCK pipeline
L4

Context & Completeness

Score input quality and gather enterprise context
Section-weighted scoring: Vision 30%, Resources 15%, Technology 15%, Required 40%. Quality gate ≥ 0.4. Ambiguity detection for "maybe", "TBD", "approximately". Generation routes bypass quality gate.
Can request NEEDS_INFO
L5

Orchestration

Build the Intelligence Plan (IPLAN) — the contract for L6
40+ use-case keys → blueprints. Engine Router resolves primary/fallback engines. Plugin Registry selects models. Output Schema Registry binds response format. IPLAN packaged with token/time/cost budgets.
L6

Reasoning — SAFE ZONE

Execute governed intelligence tasks
IPLAN LAW enforced: no Engine C, redaction required for external. Execution: Engine A → Engine B → Agents. Progressive enhancement: agent baseline + LLM enrichment. Draft artifacts created with versioning. 60s SLA / 120s timeout.
L7

Validation & HITL

Quality gate before output delivery
5 threshold checks: confidence ≥ 60, minimum 2 options, risk data present, evidence present, budget ≤ 10M AED. 3 bias detectors: single option, extreme score gap, high recommendation despite risk.
Can pause for HITL review
L8

Memory & Learning

Persist decision and trigger post-approval actions
Creates decision summary, gathers evidence, extracts rationale, generates tags. Post-approval: portfolio_sync agent executes. Engine C distillation scheduled (runs separately).

Engine A — Sovereign Internal

Local only · Zero external network calls
  • Ollama runtime (Mistral / DeepSeek / Llama)
  • RAG search via pgvector embeddings
  • Rule-based scoring & prioritization
  • Entity extraction from input
  • Historical pattern matching
  • Local LLM artifact generation
  • Policy-aware reasoning
  • Secure document analysis

Engine B — Hybrid Augmented

External LLMs via redaction gateway
  • Claude Sonnet (primary provider)
  • GPT-4o (fallback provider)
  • Structured options generation
  • Risk analysis & assessment
  • Strategic assessment
  • Executive content writing
  • Dual-provider consensus (high-value)
  • Circuit breaker per plugin

Engine C — Distillation

Post-approval only · Never runs in L6
  • Pattern extraction from decisions
  • Best practice harvesting
  • Template improvement
  • Scoring weight updates
  • Routing insight refinement
  • Policy test case generation
  • Training sample preparation
  • All outputs start as DRAFT

Security LayerMechanismBehavior
CSRF Protection attachCsrfToken + requireCsrfProtection 32-byte random token via XSRF-TOKEN cookie. Strict origin checking required in production. Double-submit pattern.
Authentication requireAuth() Validates session.userId and session.role. Returns 401 if missing. Session stored in Redis → PostgreSQL fallback.
Role Authorization requireRole(roles[]) Checks user role against allowed list. super_admin bypasses all role checks. Returns 403 on failure.
Permission Authorization requirePermission(permission) Checks effective permissions including custom overrides stored in users.customPermissions (JSONB). Hierarchy: role defaults + custom overrides.
Tenant Isolation getAuthenticatedOrganizationId() Extracts organizationId from session. All data queries automatically scoped. Optional bypassTenantCheck for system operations.

Section 17

Non-Functional Requirements

Performance SLAs, security controls, availability targets, and scalability boundaries for production deployment.

Performance

  • L6 Reasoning SLA: 60 seconds
  • L6 Hard Timeout: 120 seconds
  • Agent baseline: <1ms (zero LLM calls)
  • Demand field generation: ~33 seconds
  • Local LLM inference: 300s max
  • Health check: 3s per service
  • DB connection: 10s timeout
  • API rate limit: 2000 req/min

Security

  • CSRF: double-submit + strict origin in production
  • Session: 32-char minimum secret, Redis-backed
  • Passwords: 8-char minimum, scrypt hash
  • Rate limiting: 3 tiers (AI / upload / standard)
  • RBAC: role + custom permission overrides
  • Data masking: redaction gateway for external calls
  • Audit: all auth actions logged
  • Tenant: organizationId + RLS isolation

Availability & Resilience

  • Docker restart: unless-stopped policy
  • Health probes: 30s interval, 10s timeout, 3 retries
  • Circuit breaker: per-plugin, auto-failover
  • Session failover: Redis → PostgreSQL
  • Rate limiter failover: Redis → MemoryStore
  • DB retry: 3 attempts, exponential backoff
  • Agent baseline: always available (no external deps)
  • K8s: HPA autoscaling in production

Scalability

  • DB pool: 10 connections per instance
  • Stateless API: horizontal scaling ready
  • Worker: independent scaling via separate container
  • AI Gateway: optional, scales independently
  • Redis: centralized session/cache shared store
  • Knowledge embeddings: pgvector ANN indexing
  • K8s HPA: CPU/memory-based auto-scaling
  • Helm: environment-specific values overlays

Internationalization

  • Languages: English (en) + Arabic (ar)
  • RTL: full right-to-left layout support
  • Currency: AED (UAE Dirham) for budgets
  • Phone: international format (WhatsApp)
  • Date: locale-aware formatting
  • Content: bilingual UI with LanguageSwitcher

Sovereignty & Compliance

  • Data residency: sovereign data stays local
  • Engine A: zero external network calls
  • Redaction: PII masking before external calls
  • Classification: auto-detect gov/health/finance/defense
  • Approval: authority matrix for risk × budget
  • Audit: full decision pipeline trace
  • RLS: PostgreSQL row-level security
  • UAE gov alignment: sector-specific policies
